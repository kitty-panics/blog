Linux 下开发的一些约定俗成

为方便使用和开发，使用 Linux 时需要遵守一些约定俗成。

像 rm 不要加 -f 参数，使用 mv 替代 rm，不要日常使用 ROOT 用户，使用 ROOT 权限需通过 sudo 这些烂大街的就不说了。

配置文件

任何时候改动配置文件都应将作用域限定在自身所在的用户中，原因有二。

  1. Linux 是一个多用户操作系统，修改配置不应该影响到其它用户 (不过这一点在个人电脑上倒是无所谓，但出于保持系统文件的整洁建议遵守)
  2. 软件升级时，很可能会覆盖掉所改动的配置文件，造成配置失效。

当然也有些软件必须修改系统级配置文件才能生效，对于这种情况，在改动时应 cp 一份同名但后缀为 .bak 的文件，方便在配置文件被改得面目全非时方便还原。

包管理

为提高系统的稳定性，安装软件应通过包管理进行安装，如果官方/第三方仓库没有就自己打包。

这样做是为了不用人脑记忆软件版本、依赖关系等信息，不然在后续更新时， 可能一不小心会覆盖掉系统原有的库文件，影响到部分软件的正常运行。

  • 一个反向案例：记一次Centos 6.1云服务器升级libc-2.15.so失败过程 。

Android SDK

前面说到，应尽可能使用包管理工具安装一切软件，不过对于 Android SDK 是个例外。

虽然有的发行版官方或第三方软件仓库会提供 Android SDK 的包，但不建议使用，因为 Android SDK 的包一般会被安装到 /opt/android-sdk 目录，使用 SDK Manager 需要 ROOT 权限， 会有诸多不便。而应该由 Android-Studio 来管理 SDK，SDK 安装路径也应该设置在用户目录下。

Python / Node.js

这两个软件合在一起说。

如果官方或第三方软件仓库没有想要的 Python / Node.js 的模块，也不建议使用 sudo pip install 和 sudo npm -g install 进行安装。因为这会将模块安装到系统目录，造成系统的包管理处理对应文件时出现错误。 再加上 pip 和 npm 卸载时不会一并卸载依赖，会残留很多文件在系统目录，破坏系统文件的整洁。

正确的做法是将 $HOME/.local/bin 和 $HOME/.node_modules/bin 加入到 PATH 中，再通过 pip install --user 和 npm -g install 安装，当然 Python 更推荐使用 Virtual environment 。

如果必须将 Python 和 Node.js 的模块安装到系统目录，应自行为所使用的发行版打包。
