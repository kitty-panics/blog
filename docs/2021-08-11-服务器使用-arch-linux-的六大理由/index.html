本来打算写一篇有关 Linux 服务器安全配置的博文，但网上已经有很多相关的文章了， 而且服务器安全这块最近似乎也没出现什么新鲜的玩意。

那就借 Arch Linux 之名水一篇博客，顺带抨击一些部分人奉为圭臬的谬论。

一、更易理清配置

各云服务商提供的系统都包含一些定制的配置，而这些配置在不断变动，还不告诉你动了啥。加上很多人安装/配置软件， 喜欢用网上的一键脚本，安不安全另说，这会让本来就难以理清的配置变得更加混乱。

记得之前在某云开了台服务器跑个服务，一天后 发现性能不够，又开了台相同地区、同样系统的机子做负载均衡， 结果新开的这台部署完后发现无法访问，开始还以为是自己的问题，排查后发现多了个 UFW ，外加一百多行乱七八糟的规则。 如果还使用了一键脚本进行部署，还得检查脚本是否写的有问题，更是加大了工作量。

对于这种黑箱系统，换成 Arch Linux 这类做加法、遵循上游的发行版是一个很好的选择。

二、线上线下环境一致

各发行版包管理器不同，打包粗细粒度不同，有些包名字也不同，有的发行版有的包而另一发行版却又没有， 甚至同一发行版相同版本下不同云服务商提供的包也不同。

所以使用 Arch Linux 这类线上线下皆宜的发行版，本地怎么配置，线上就怎么配置，不用纠结发行版间的差异。

基于以上两点，所以 Docker 才是最大的赢家？

三、更安全？

虽然说系统安全得益于尽可能牢固的安全配置，但也需要运维人员有足够的安全意识，有部分人会为了稳定性不升级 (也可能如第五点中那样，压根无从升级)，虽然系统一直不升级能用安全性换取稳定性。但这样的稳定如空中楼阁， 因为很大部分网络攻击就是针对旧版本软件中的漏洞，一个频繁被攻击甚至被攻破的系统，有何稳定可言？

而 Arch Linux 用户普遍有更新强迫症，能及时更新到新版，能有效避免很多漏洞扫描和攻击。

另外，看过一些恶意脚本，有部分会针对 deb/rpm 系发行版优化，使用 Arch Linux 或许能逃过一劫。

  • CentOS: 永远有多远就离它多远
  • 反向案例：晒晒你的vps开机时间

四、更早体验到新功能

Debian 系的软件太老了，CentOS 更老 (还好它 死了，可惜又 没完全死)。想体验一些新特性， 得等到候年马月。

Arch Linux 是一个滚动式更新发行版，不断更新软件听起来好像很不稳定的样子，但前面说到 Arch Linux 是一个遵循上游的发行版，稳定版仓库的包就是上游释出稳定版后打出来的，这和去官网下载稳定版安装没有任何区别。

诚然软件新版本会有新的 BUG，但同时也修复了旧的 BUG 和带来新的特性，利远大于弊。像 JDK、Python 之类的软件因需求需要停留在低版本也是可以的。

所以使用 Arch Linux 这类滚动式更新 (我称之为渐进式升级) 的发行版会有一个很棒的体验。

五、打包简单

一直不太理解的是，有的人一边义正辞严的高呼服务器用 Arch Linux 不稳定，一边孜孜不倦的用 ./configure && make && make install 安装软件是什么心理。

从便捷性和稳定性考虑，所有软件都应通过包管理器进行安装，如果有的软件没包？那就自己打包。

手动编译安装意味着要手动处理软件依赖关系，很可能会不小心覆盖掉系统原有的动态库文件， 从而影响到部分软件的正常运行。如果编译安装的软件过多，依赖关系变的复杂，加上时间一长， 在后续升级和修补漏洞时连自己都忘了需要处理哪些依赖，这样做严重破坏了系统的稳定性。

  • 反向案例：记一次Centos 6.1云服务器升级libc-2.15.so失败过程 。

所以要是喜欢编译的话，可以使用 Arch Linux 这种打包系统简单的发行版，当然 Gentoo 也是一个不错的选择。

六、用户贡献度高

用其它主流发行版时，不得不说资料是真的多，教程一搜一大把，但大部份都是重复转载。 很多教程都在教你要先安装哪些依赖包，编译参数要加啥。可这些编译源码一两次就塾悉了， 万变不离其宗。

唯独没人打个包供下载。

而 Arch Linux 的 AUR 中，有用户贡献的近 8 万个软件打包脚本。不想编译的话还有很多 个人源 供选择，我也维护了一个 个人软件源 。
